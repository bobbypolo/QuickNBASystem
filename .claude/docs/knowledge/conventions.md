# Naming Conventions — Single Source of Truth

## Requirement ID Pattern (R-PN-NN)

**Canonical regex**: `R-P\d+-\d{2}`

**Format**: `R-P{phase}-{sequence}` where:

- `R` = Requirement prefix (constant)
- `P{phase}` = Phase number (1-based integer, no zero-padding)
- `{sequence}` = Two-digit zero-padded sequence within the phase (01-99)

**Examples**: R-P1-01, R-P1-08, R-P3-12, R-P6-09

**Used by**:

- `PLAN.md` Done When sections — defines each requirement
- `prd.json` acceptanceCriteria[].id — links stories to requirements
- Test files — `# Tests R-PN-NN` marker in docstrings for traceability
- `verification-log.md` — `R-PN-NN: PASS/FAIL (evidence: ...)` format
- `/audit` Sections 3 and 8 — greps for markers to validate traceability chain

## Traceability Chain

```
PLAN.md "R-P1-01: [criterion]"
    ↓ (extracted by /plan Step 7)
prd.json { "id": "R-P1-01", "criterion": "...", "testType": "..." }
    ↓ (enforced by Builder TDD)
test file: # Tests R-P1-01
    ↓ (validated by QA Step 11)
verification-log.md: R-P1-01: PASS (evidence: test_name passed)
    ↓ (audited by /audit Section 3)
Full traceability: requirement → story → test → verification
```

## Story ID Pattern

**Format**: `STORY-{sequence:03d}` (e.g., STORY-001, STORY-006)

Auto-generated by `/plan` Step 7 from PLAN.md phase numbers.

## Branch Naming

**Ralph feature branches**: `ralph/{plan-name}` (e.g., `ralph/v-model-upgrade`)

Derived from the plan name in PLAN.md title or user input during Ralph STEP 1.5.

## Commit Message Convention

**Format**: Conventional Commits (`feat:`, `fix:`, `docs:`, `chore:`)

**Ralph story commits**: `feat({story.id}): {story.description}`

Example: `feat(STORY-001): prd.json v2 schema template`

## Phase Type Convention

**Field**: `phase_type` in prd.json story objects

**Valid values**:

| Value         | Description                                     | QA Behavior                                |
| ------------- | ----------------------------------------------- | ------------------------------------------ |
| `foundation`  | Core infrastructure, utilities, base modules    | Skips integration tests (Step 4)           |
| `module`      | Self-contained feature modules                  | Runs all steps except E2E                  |
| `integration` | Cross-module wiring, API boundaries             | All steps run; integration tests mandatory |
| `e2e`         | Full system paths, deployment, end-to-end flows | All steps run; most rigorous               |

**Used by**:

- `prd.json` — each story can specify a `phase_type` for adaptive QA
- `qa_runner.py` — `--phase-type` argument controls which QA steps are skipped
- `/plan` Step 7 — auto-assigns phase_type based on plan phase characteristics

**Default**: If not specified, all QA steps run (equivalent to `e2e`).
